# v0.4 Candidates

Items considered during v0.3 planning and deliberately deferred. Each
entry includes the reason for deferral and a rough sizing.

## Asymmetric signature schemes

- **RSA-SHA256** — common in Apple webhooks and some enterprise
  providers. Requires PEM key parsing and a new config field
  (`publicKeyEnvVar`) that doesn't fit the symmetric `secretEnvVar`
  model. Deferred because no concrete user request yet; ~150 LOC.
- **ed25519** — used by some niche providers. Similar design
  considerations as RSA but with `System.Security.Cryptography.Ed25519`
  (or the `NSec.Cryptography` NuGet for older runtimes). Deferred
  alongside RSA; ~150 LOC.
- **ECDSA-SHA256** — Apple Server Notification v2 uses it. Same
  design footprint as RSA. Deferred until there is user demand.

Recommendation: design as a new `AsymmetricSignatureScheme`
implementing `IIngestSignatureScheme`, accepting `publicKeyEnvVar`
alongside / instead of `secretEnvVar`.

## Postgres bytes-body migration support

The `00000000000001_BytesBodyAndArrayHeaders` migration throws
`NotSupportedException` on non-SQLite providers. If Postgres adoption
grows, port the recreate-and-copy logic using PostgreSQL's
`ALTER TABLE ... USING convert_to(...)` pattern. Deferred in v0.3
because Postgres users on v0.1 must already recreate the DB; the
migration path is not yet tested against a real Postgres deployment.

## Auth auto-detection by bind address

`HOOKVAULT_NO_AUTH=true` is a coarse opt-out. A smarter mode would
detect whether Kestrel is bound to `127.0.0.1` only vs `0.0.0.0` and
skip auth automatically when localhost-only. Deferred because the
detection logic adds complexity (Kestrel endpoint config is not
trivially inspectable post-build) that wasn't worth it for v0.3's
single-flag solution.

## Fixtures library

A "saved payloads" feature: capture a webhook, save it as a named
fixture, then replay any fixture against any provider's `forwardUrl`
on demand. Out of scope for v0.3's minimal body-edit decision. Worth
revisiting if users start asking for fixture re-use across projects or
test suites.

## `armv7` Docker images

Raspberry Pi 32-bit support. Skipped from v0.3 multi-arch (which
covers `linux/amd64,linux/arm64`) due to diminishing-returns audience
and significantly slower QEMU cross-build times.

## In-UI body editor improvements

The v0.3 body-edit panel is a plain `<textarea>`. Potential upgrades:

- JSON syntax highlighting (Monaco or CodeMirror — larger bundle).
- JSON validation + format-on-paste.
- Diff view: side-by-side original vs edited body.
- History: persist the last N edited bodies per event for re-use.

Deferred because the plain textarea covers the primary use case and
the additional bundle weight was not justified for v0.3.

## Tracing (OTLP)

v0.3 ships metrics only. If users deploy HookVault behind a
distributed tracing collector, adding an OTLP exporter with spans
around ingest + forward + replay would complete the observability
picture. Deferred until there is concrete demand; metrics provide
sufficient operational visibility for local dev.

## Webhook source IP allowlists per provider

Today HookVault accepts ingest from any source IP. Some providers
publish known source IP ranges (e.g. Stripe's webhook IP list). A
per-provider `allowedSourceIps` field would let HookVault reject
spoofed deliveries before signature validation — defence in depth.
Deferred because the target environment is local dev (trusted network)
and maintaining provider IP lists adds ongoing operational burden.
