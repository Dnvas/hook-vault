# HookVault v0.3 — Polish, Ergonomics, Distribution

**Date:** 2026-05-16
**Status:** Design — pending user approval
**Author:** Brainstormed via `superpowers:brainstorming`
**Baseline:** `v0.2.0` (`ghcr.io/dnvas/hookvault:v0.2.0`)

## Goal

Address the rough edges from the post-v0.2 retrospective and ship the
items deferred-by-design during the hardening sprint. The release closes
the gap between "technically complete" and "polished v1.0-ready" without
expanding the spec into a new product direction.

## Strategic decisions (locked in)

| Decision | Choice | Rationale |
|---|---|---|
| Auth model | **Opt-out via `HOOKVAULT_NO_AUTH=true`** | Single-user local dev: friction wins over defense-in-depth. Loud startup warning when disabled. Default behaviour unchanged. |
| In-UI body edit | **Minimal: edit-and-replay one event** | Solves the killer dev case (tweak a captured payload, fire it at your local app) without the scope of a fixtures library. |
| PR shape | **Three PRs: polish, ergonomics, distribution** | Each is independently reviewable. PR1 is small (~250 LOC), PR2 is the largest (UI work), PR3 is mostly ops/docs. |
| Asymmetric crypto (RSA-SHA256, ed25519) | **Deferred to v0.4** | Don't fit the `secretEnvVar` symmetric-key model; need PEM parsing + new config field. Out of scope here. |
| Postgres bytes-body migration | **Won't do** | Zero existing users with Postgres data to migrate; the current `NotSupportedException` guard is fine forever. |
| Fixtures library | **Won't do** | Out-of-scope per "minimal" body-edit decision. Possible v0.4 spec if friction surfaces. |

## Architecture overview

Three orthogonal change streams; each maps to one PR.

1. **Polish** — touches `Program.cs` auth wiring, `EventRepository` /
   `EventsController` for pagination, `EventRetentionWorker` +
   `HealthController` for the retention surfacing. No new files apart
   from `RetentionStats`. Small.
2. **Ergonomics** — adds `ProviderConfig.CaptureOnly`, extends the
   replay endpoint with an optional body override, adds the
   `hmac-sha1` algorithm case, lands the UI body-edit flow. Touches
   the React UI for the first time since Phase 6.
3. **Distribution** — adds OpenTelemetry + Prometheus exporter packages,
   wires `/metrics`, extends `release.yml` for `linux/arm64`, updates
   every agent-facing doc (`hookvault-spec` skill, `README`,
   `SECURITY`, `CHANGELOG`).

## PR 1 — Polish

Small, safe, sets up the v0.3 baseline. ~250 LOC.

### 1.1 Auth opt-out

`HOOKVAULT_NO_AUTH=true` disables `[Authorize]` enforcement. The flag is
read once at startup. Implementation: when set, register a permissive
authorization default policy that always succeeds; the `JwtBearer` scheme
stays registered so the SSE token-via-query-param path still works for
any caller that *does* present a token. Loud `LogWarning` line on
startup describing the implication.

When the flag is unset, behaviour is identical to v0.2.

The startup log line:

```
[Startup] HOOKVAULT_NO_AUTH=true: the management API is unauthenticated.
Anyone who can reach this listener can read, replay, and delete events.
Make sure HookVault is bound to 127.0.0.1 only, or remove this flag.
```

### 1.2 Body-search pagination correctness

When `bodyContains` is set, `ListSummariesAsync` falls back to in-memory
post-filtering and the SQL `total` count is pre-filter — currently
misleading. Fix:

- Response shape gains a new field `totalApproximate: bool` (defaults
  to `false`).
- When `bodyContains` is set, `total: null` and `totalApproximate: true`.
- TypeScript `ListEventsResponse` updated.
- UI hides numeric pagination when `totalApproximate`; renders "N events
  shown — refine your query for an exact count".

API contract is back-compat (additive field; existing clients ignore it).

### 1.3 Retention sweep surfacing

`EventRetentionWorker` currently logs sweep results but doesn't expose
them. Add a `RetentionStats` singleton:

```csharp
public sealed class RetentionStats
{
    public int? MaxEvents { get; init; }
    public TimeSpan? Retention { get; init; }
    public DateTimeOffset? LastSweepAt { get; private set; }
    public int LastSweepDeleted { get; private set; }

    public void RecordSweep(int deleted) { /* lock-protected write */ }
}
```

`EventRetentionWorker` calls `stats.RecordSweep(deletedCount)` after each
run. `HealthController` includes the stats in the response under a new
`retention` field:

```json
{
  "status": "healthy",
  "version": "1.0.0",
  "providers": ["stripe", "github", "resend"],
  "database": "sqlite",
  "eventCount": 142,
  "oldestEvent": "...",
  "retention": {
    "maxEvents": 10000,
    "retentionDays": 7,
    "lastSweepAt": "2026-05-16T13:00:00Z",
    "lastSweepDeleted": 3
  }
}
```

When the worker is idle (neither env var set), `retention` is `null`.

### 1.4 PR 1 testing

- Unit test on `RetentionStats` thread safety (two concurrent writers).
- Integration test on health response shape with retention configured.
- Integration test confirming `HOOKVAULT_NO_AUTH=true` allows
  unauthenticated `GET /api/events`.
- Integration test confirming default behaviour still requires JWT.

## PR 2 — Ergonomics

The largest piece. Backend changes are surgical; the UI work is the bulk.
~500-700 LOC.

### 2.1 Capture-only mode

New per-provider option `captureOnly: bool` (default `false`). When
`true`, `IngestController` persists the event with the new
`EventStatus.Captured` value and **does not call**
`forwarder.ForwardAsync`. The event is visible in the UI; users can
replay it manually whenever their local app comes up.

A new enum value (rather than reusing `Received`) keeps the lifecycle
honest — `Received` is a transient state on the path to `Forwarded` /
`ForwardFailed`, whereas `Captured` is a resting state by design.
Status conversion to string is already in place
(`HasConversion<string>()`), so no DB migration is needed beyond the
new enum constant.

Implementation:

```csharp
// In IngestController.Ingest, after notifier.Notify(...)
if (config.CaptureOnly)
{
    evt.Status = EventStatus.Captured;
    await repo.UpdateAsync(evt, ct);
}
else
{
    await forwarder.ForwardAsync(evt, ct);
}
```

UI changes: `EventRow` adds a slate-blue status dot for `Captured`;
`EventList` adds a `Captured` filter pill; `EventDetail` shows a small
"capture only" badge in the header so the mode is obvious.

Replay still works: clicking Replay enqueues to `ReplayWorker`, which
calls `EventForwarder.SendAsync` regardless of `CaptureOnly`.

### 2.2 In-UI body edit + replay

Two-part change:

**Backend (~80 LOC + tests):**

Existing endpoint `POST /api/events/{id}/replay` (no body). Extend to
accept an optional JSON body:

```json
{
  "body": "<edited text body>"
}
```

When the request body is present and `body` is non-null, the replay path
uses that text instead of the stored bytes. Implementation:

- `EventsController.Replay` parses optional body, passes through to a
  new `ReplayQueue.EnqueueWithBodyAsync(id, body, ct)` overload.
- `ReplayQueue` now ships `ReplayJob` instead of bare `Guid`:
  `record ReplayJob(Guid EventId, byte[]? BodyOverride)`.
- `ReplayWorker.ProcessAsync` reads the override; when present, passes
  the overridden bytes to `EventForwarder.SendAsync` via a new
  overload `SendAsync(WebhookEvent evt, byte[] body, CancellationToken ct)`.
  The original `SendAsync(evt, ct)` becomes a thin wrapper around the
  new method using `evt.Body`.
- The stored event is **not mutated**. `ReplayCount` still increments;
  `LastReplayAt` still updates. A new field `LastReplayWithEditedBody:
  bool` flags whether the most recent replay used a custom body — purely
  informational for the UI.

**Frontend (~250 LOC + a new component):**

Detail panel adds an "Edit & Replay" button next to the existing
"Replay" button. Click opens an inline editor (textarea-based, not a
full code editor — minimal scope):

- Pre-populated with the current body text.
- Two buttons: "Replay with edits" (submits) and "Cancel" (collapses).
- On submit: `POST /api/events/{id}/replay` with the edited body in the
  request JSON. Optimistically marks the row as "Replaying" until the
  SSE notification updates the status.
- Editor uses Tailwind monospace; no syntax highlighting, no JSON
  formatter — keep it minimal.

The button only appears when the event has a non-empty body.

### 2.3 HMAC-SHA1

`SingleHeaderHmacScheme` algorithm switch gains `hmac-sha1` →
`HMACSHA1.HashData`. Adds one line + one test. Update the GitHub example
config doc-comment to mention SHA-1 is now supported for legacy
webhooks.

### 2.4 PR 2 testing

- New `CaptureOnlyTests`: open provider with `captureOnly: true`
  returns 202, event status `Received`, forwarder never invoked.
- New `BodyEditReplayTests`: replay with custom body forwards the
  override, not the stored body. Real `CapturingHandler` asserts the
  forwarded bytes.
- New `HmacSha1Tests`: valid SHA-1 signature passes, tampered fails.
- New `tests/e2e/body-edit.spec.ts` scenario file describing the UI
  flow (consistent with existing e2e specs).

## PR 3 — Distribution + docs

Smaller code than PR2, more docs. ~300 LOC + many docs.

### 3.1 Prometheus `/metrics` endpoint

Add packages:

```xml
<PackageReference Include="OpenTelemetry.Extensions.Hosting" Version="1.10.0" />
<PackageReference Include="OpenTelemetry.Exporter.Prometheus.AspNetCore" Version="1.10.0-beta.1" />
<PackageReference Include="OpenTelemetry.Instrumentation.AspNetCore" Version="1.10.1" />
```

Wire up:

```csharp
builder.Services.AddOpenTelemetry()
    .WithMetrics(b => b
        .AddAspNetCoreInstrumentation()
        .AddMeter("HookVault")
        .AddPrometheusExporter());
```

```csharp
app.MapPrometheusScrapingEndpoint("/metrics");
```

The `/metrics` endpoint is **unauthenticated** (operational data, not
secrets) and lives outside the `[Authorize]`-gated controllers. Bound
under the same listener as the rest of HookVault.

Custom metrics published via a new `HookVaultMeter` singleton:

| Metric | Type | Labels | Description |
|---|---|---|---|
| `hookvault_events_total` | Counter | provider, status | Total events captured |
| `hookvault_replays_total` | Counter | outcome (success/failed/exhausted) | Replays attempted |
| `hookvault_forward_duration_seconds` | Histogram | provider, outcome | Forward call latency |
| `hookvault_retention_deleted_total` | Counter | reason (count_cap/age) | Events deleted by retention worker |
| `hookvault_signature_validation_total` | Counter | provider, result (valid/invalid/skipped) | Signature checks |

The standard `aspnetcore.routing.match_attempts`, `http.server.request.duration`,
`kestrel.active_connections`, etc. come for free from
`AddAspNetCoreInstrumentation`.

    platforms: linux/arm64

- name: Build and push
  uses: docker/build-push-action@v7

QEMU emulation for arm64 builds is slow (~5-8 min total) but works. Skip
armv7 — the diminishing-returns audience isn't worth the build time.

## Risks

- **`ReplayQueue` channel type change.** `Channel<Guid>` → `Channel<ReplayJob>` is a breaking internal API change; existing replay tests need updating. The replay endpoint signature stays the same; the queue contract is implementation detail.
- **OpenTelemetry beta package.** `OpenTelemetry.Exporter.Prometheus.AspNetCore` is officially in beta. If it churns in a future release this needs a NuGet bump. The stable alternative is `prometheus-net.AspNetCore` (different API). Accept the beta risk for now — it's been beta for a long time and is widely used.
- **`HOOKVAULT_NO_AUTH` foot-gun.** If a user enables this and then ports HookVault to a non-local environment, they expose everything. The loud startup warning is the only mitigation. We document in SECURITY.md.
- **arm64 build time.** QEMU emulation can push CI runtime to ~10 minutes per release. Tolerable for tag-fired releases (low frequency) but worth tracking.

## Out of scope

- Asymmetric crypto schemes (RSA, ed25519, ECDSA) — v0.4 if needed.
- Saved-payloads library / fixtures store — v0.4 candidate.
- Auth modes beyond JWT + opt-out (OIDC, basic auth, etc.) — not needed for the dev-tool threat model.
- Distributed tracing (OTLP exporter) — only metrics in v0.3. Tracing is a follow-up if any user actually deploys HookVault behind a tracing collector.
- `armv7` Docker images.
